## 1. 分支规范

主分支：

```bash
main
```

不要直接在 `main` 上写功能。

每个功能新建一个分支：

```bash
git checkout -b feature/static-pages
git checkout -b feature/posts-api
git checkout -b feature/mysql-posts
```

常见分支命名：

```bash
feature/xxx   # 新功能
fix/xxx       # 修 bug
docs/xxx      # 文档修改
style/xxx     # 样式调整
refactor/xxx  # 代码重构
```

例如：

```bash
feature/homepage
feature/admin-login
fix/post-detail-error
docs/update-readme
```

## 2. Commit 信息规范

推荐格式：

```bash
类型: 本次提交做了什么
```

常用类型：

```bash
feat: 新功能
fix: 修复 bug
docs: 修改文档
style: 修改样式，不影响逻辑
refactor: 重构代码
chore: 配置、依赖、杂项
init: 初始化项目
```

例子：

```bash
git commit -m "init: 初始化项目结构"

git commit -m "feat: 完成首页静态页面"

git commit -m "feat: 添加博客列表和详情页"

git commit -m "feat: 接入文章列表 API"

git commit -m "fix: 修复博客详情页刷新后数据丢失问题"

git commit -m "docs: 更新项目运行说明"

git commit -m "style: 优化博客卡片样式"

git commit -m "chore: 配置 .gitignore"
```

## 3. 一次 commit 只做一件事

不要这样：

```bash
git commit -m "改了一堆东西"
```

也不要一个 commit 里同时做：

```text
登录功能 + 删除文章 + 修改 README + 改样式
```

应该拆开：

```bash
git commit -m "feat: 添加管理员登录页面"
git commit -m "feat: 实现文章删除接口"
git commit -m "docs: 补充后台功能说明"
git commit -m "style: 调整后台表格样式"
```

## 4. 提交前固定流程

每次提交前都走一遍：

```bash
git status
git diff
git add .
git commit -m "feat: 完成某某功能"
```

如果用 TortoiseGit，就看：

```text
右键项目目录
→ Git Commit
→ 先检查 changed files
→ 确认没把 .env、node_modules 提交进去
→ 写 commit message
→ Commit
```

尤其要注意这些**不要提交**：

```bash
node_modules/
dist/
.env
.DS_Store
*.log
```

## 6. 给学生的最简版要求

你可以直接这样要求他们：

```text
1. 不允许直接在 main 分支开发。
2. 每个阶段新建一个 feature 分支。
3. 每完成一个阶段至少提交一次 commit。
4. commit 信息必须写清楚做了什么。
5. 提交前必须检查改动文件。
6. .env、node_modules、dist 不能提交。
```

最简单命令模板：

```bash
git checkout -b feature/功能名
git status
git add .
git commit -m "feat: 完成某某功能"
git push origin feature/功能名
```

对新手来说，先不必太复杂，能做到“**分支清楚、提交清楚、不乱传敏感文件**”就够了。
这个是git提交的规范，现在也不严格要求，但是还是需要了解一下，自己开发的话没有这么看重，但是如果接触的是大项目或者多人协作的项目一定要做好git的规范，否则会导致项目比较混乱，哪里有bug也不好回溯
git就是一个版本管理工具，你的每一次提交都会有记录，改了哪些文件都很清晰，比如你不小心误删了某个文件，但是你前面在git已经提交过了，就可以很方便的找回来
.gitignore是git的配置文件，他负责告诉git哪些文件应该忽略掉，不应该提交上去，例如图中的node_modules，这个就是项目下载的依赖文件，这些文件很多很大并且不是必要的，所以会忽略他，同时还有类似于.env、.comfig这种环境配置文件也应该忽略，这种配置文件中很可能会存储一些敏感数据，如果是自己的私有仓库还好，但如果是公开仓库这些信息上传了很有可能就会被别人窃取，造成安全问题
